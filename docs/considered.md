# Considered but skipped or deferred

Models that were evaluated for inclusion and intentionally left out (or deferred to a later rig pass), with the reason. The bar lives in `harness.py`'s scope: must install cleanly cross-platform in a self-contained venv, fit the runner protocol (text in → wav out, single process), and have a license that permits redistribution of benchmark results. CPU support is preferred but no longer required as of 2026-05-24 — CUDA-only models can be added if they fail gracefully on non-CUDA rigs.

A 2026-05-24 update added a Linux-3090 rig to the bench. Several entries below were originally skipped because of Windows-only blockers (vLLM Blackwell wheels, Linux-only compiled extensions, etc.); those become re-evaluable on the Linux rig and are flagged below where applicable.

## Future enhancement (queued, not yet implemented)

- **[VibeVoice-Large 9B](https://huggingface.co/aoi-ot/VibeVoice-Large)** (Microsoft's 9B variant, redistributed under MIT via `aoi-ot/VibeVoice-Large` and `vibevoice/VibeVoice-7B` — same weights, two naming conventions: "7B" labels by Qwen2.5-7B LLM backbone count, "9B" by total params including audio components). ~20-24 GB VRAM at fp16. Fits Win-5090 (32 GB) comfortably, **borderline on Linux-3090 (24 GB)** — KV cache for our 100+ token prompts may push it over. Fails Mac M4 (16 GB). For now we ship VibeVoice-1.5B (3B total, ~7 GB VRAM) as the cross-rig mid-tier addition; Large 9B is the quality-ceiling future enhancement, gated on either (a) confirming the 3090 fits the model with our prompts or (b) shipping a 4-bit quantized GGUF variant (community has `FluffyBunnies/vibevoice-onnx-int4` and `callgg/vibevoice-bf16` as candidates).

- **[Fish Audio S2 / S2-Pro](https://github.com/fishaudio/fish-speech)** (current `main` branch, 4B params). **Reclassified 2026-05-24 from skipped to queued for the Linux-3090 pass.** Prior block was "Linux/WSL only per official docs; Windows native unsupported" — that's resolved on a native Linux rig. 24 GB VRAM floor barely fits the 3090. Still needs a Python wrapper to orchestrate the 3-stage CLI pipeline (DAC encode → text2semantic → DAC decode) so the runner sees `text in → wav out`. Wrapper is ~1-2 days of work. Will run on Linux-3090 only; Win + Mac rows show fail (documented). Research-license non-commercial (benchmark redistribution OK).

- **[MOSS-TTS-Realtime](https://github.com/OpenMOSS/MOSS-TTS)** (OpenMOSS, 1.7B, `MossTTSRealtime` architecture, Apache 2.0). Streaming/real-time TTS for low-latency voice agents — upstream advertises a TTFB of 180 ms via incremental synthesis with multi-turn context. Needs a streaming-bench lens (sub-sentence latency, time-to-first-chunk, mean inter-chunk gap) which our current harness doesn't measure. A non-streaming wrapper would understate the model's design point — the 180 ms claim only shows up when the harness watches the first audio frame rather than the full tensor. Queued behind the "streaming bench lens" future enhancement (see [tasks.md](tasks.md)). The non-realtime siblings MOSS-TTS-Nano (100M) + MOSS-TTS flagship (8B) are in the harness as of 2026-05-25.

- **[MOSS-TTSD / MOSS-VoiceGenerator / MOSS-SoundEffect](https://github.com/OpenMOSS/MOSS-TTS)** (OpenMOSS, 1.7B-8B, Apache 2.0). Three sibling models in the MOSS-TTS family with different task shapes:
  - **MOSS-TTSD** is a *dialogue* model (multi-speaker, ultra-long dialogues). The harness's `(text in → wav out)` contract assumes single-speaker monologue prompts; benching TTSD against single-speaker models would compare apples to oranges. Closer fit: a Dia-style "dialogue bench" track once one exists.
  - **MOSS-VoiceGenerator** is a *voice design* model — generates voices from text prompts without any reference speech. Task shape is "describe a voice → produce a speaker embedding/wav", orthogonal to either predefined or cloning TTS.
  - **MOSS-SoundEffect** generates sound effects, not speech. Out of scope for a TTS bench.
  All three deferred until a separate task-shape track exists for each.

- **[Zonos2](https://github.com/Zyphra/ZONOS2)** (Zyphra, Apache-2.0, `Zyphra/ZONOS2`, 44.1 kHz). **Surfaced 2026-07-21 from the [mahimairaja/tts](https://github.com/mahimairaja/tts) "Awesome TTS" list; recommended-to-add, gated only on the runner build (not on any missing bench feature).** Successor to our `zonos` (Zonos v0.1) — a sparse MoE at ~900M active / 8B total, multilingual zero-shot cloning plus bundled default voices. Passes every inclusion bar cleanly: **Apache-2.0** (cleaner than nothing-to-fix but a fresh permissive grant), fits Linux-3090 (24 GB) and Win-5090 (32 GB) at bf16 (~16 GB), and — unlike `higgs_v3` / `fish_s2` / `miotts` — ships a genuine **in-process Python API** (`tts.generate_one(text, TTSSamplingParams(), language=..., speed=...)` → audio) that drops straight into the `text in → wav out` runner protocol with **no server or subprocess wrapper**. One constraint from its own README: **"Linux only (x86_64), NVIDIA CUDA required"** — so it's a **Linux-3090 add with Win/Mac fail rows** (same pattern as dots.tts / Voxtral-cloning), installed via `git clone && uv sync` (source clone, not a pip wheel). Effectively **supersedes the Zonos v0.1 row**. Effort: low–moderate (thin runner around `generate_one` + install stanza). **Status 2026-07-21 (branch `zonos2-add`, not yet on master):** `runners/zonos2_runner.py` + `harness.py` row + full registration (report.py's 9 registries, NO_PRESET_VOICE, README/cloning/known-issues, `install.sh` stanza) all DONE and test-verified (52 pytest pass + registry-drift guard green). Runner API confirmed vs upstream commit `194c0a3` but **not yet executed** — remaining = install (`git clone && uv sync` + CUDA kernel build) + smoke (canonical prompts) + bench + scoring on the **Linux-3090**.

_Implemented & live (no longer queued):_ **LFM2.5-Audio-1.5B** — added to the harness as `lfm2_audio` and **published to the default board 2026-06-22** (predefined-voice / `kind=predefined`; its TTS mode offers 4 preset voices, verified no cloning; 24 kHz; English-only; single-process `pip install liquid-audio`, cross-rig cpu+cuda). See `runners/lfm2_audio_runner.py` + `docs/known-issues.md`.

_Implemented & live (no longer queued):_ **MioTTS 0.1B + 0.6B** — added to the harness as `miotts_01b` / `miotts_06b` (server-backed: a llama.cpp/Ollama LLM emits MioCodec tokens → the MioTTS REST server decodes to 44.1 kHz wav; thin HTTP-client runner `runners/miotts_runner.py` à la `higgs_v3`). **Benched + full-published + scored from the Linux-3090 on 2026-06-23** (cloning board / `kind=cloning` — pure zero-shot cloner, in `NO_PRESET_VOICE`; EN/JA, English first-class). Scores: **0.1B** clean (UTMOS 3.6–4.4, WER ≤0.18, SIM ~0.36–0.38); **0.6B** mostly good but **p1 cold over-generated to ~17 s on the short prompt → UTMOS 1.34 / WER 0.43 / SIM 0.26** — LLM-codec length instability, model-inherent (cf. cosyvoice p1), kept visible on purpose so the board reflects its real, sometimes-bad quality. Licenses: inference code MIT; 0.1B Falcon-LLM, 0.6B Apache-2.0. See `runners/miotts_runner.py` + `docs/known-issues.md`.

## Runtime / tooling references (not bench models — watch list)

These aren't models to add; they're native **C++/ggml runtimes** (a "llama.cpp for audio") that reimplement several models we already track. Relevance to the bench is narrow and singular: a future **native-backend speed ceiling** for the Speed lens (the same weights running Nx faster than the Python reference path we currently time), and a possible way to cut per-model Python-env install pain. They do **not** add new models and do **not** change quality scores (same weights, no quant by default), so adopting one is a methodology call, not a new row. Both are also early/solo-maintained on a forked ggml. Revisit when streaming + a real server + stable CUDA land. Logged 2026-06-26 from an r/LocalLLaMA thread.

- **[audio.cpp](https://github.com/0xShug0/audio.cpp)** (0xShug0) — **TTS-first**. 12 of 25 model families released, overlapping our `pocket`, `chatterbox`, `omnivoice`, `voxcpm`, `qwentts`, `miotts`. Claims 3–5× vs the Python reference path on CUDA (no-quant fp16/fp32; warm-session RTFx up to ~48× real time on PocketTTS long-form). Caveats: no framework-wide streaming (offline only), self-described "naive" server, GGUF support pending, CPU barely optimized (author is CUDA-focused), and Windows French PocketTTS is broken (24-layer flow mismatch). Built clean-slate around a uniform module abstraction (every op, even ADD, is a metadata-carrying module) for a planned future feature.
- **[CrispASR](https://github.com/CrispStrobe/CrispASR)** (CrispStrobe, 349★) — **ASR-first** (Parakeet TDT, Canary 1B v2, Voxtral, Cohere Transcribe + universal forced alignment) but also does TTS (Qwen3-TTS, Orpheus, **IndexTTS**). **GGUF-native** and far more mature: WASM/iOS/Android/Flutter/Go/Python bindings, HF Space demo, Docker CUDA+Vulkan, AMD/HIP, ~71 runtimes. Overlaps our `indextts` (the by-ear cloning pick) + `qwentts`. The more deployment-ready of the two; the audio.cpp author started fresh rather than contribute here specifically to avoid retrofitting his module abstraction onto CrispASR's pragmatic structure.
