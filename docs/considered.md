# Considered but skipped or deferred

Models that were evaluated for inclusion and intentionally left out (or deferred to a later rig pass), with the reason. The bar lives in `harness.py`'s scope: must install cleanly cross-platform in a self-contained venv, fit the runner protocol (text in → wav out, single process), and have a license that permits redistribution of benchmark results. CPU support is preferred but no longer required as of 2026-05-24 — CUDA-only models can be added if they fail gracefully on non-CUDA rigs.

A 2026-05-24 update added a Linux-3090 rig to the bench. Several entries below were originally skipped because of Windows-only blockers (vLLM Blackwell wheels, Linux-only compiled extensions, etc.); those become re-evaluable on the Linux rig and are flagged below where applicable.

## Future enhancement (queued, not yet implemented)

- **[VibeVoice-Large 9B](https://huggingface.co/aoi-ot/VibeVoice-Large)** (Microsoft's 9B variant, redistributed under MIT via `aoi-ot/VibeVoice-Large` and `vibevoice/VibeVoice-7B` — same weights, two naming conventions: "7B" labels by Qwen2.5-7B LLM backbone count, "9B" by total params including audio components). ~20-24 GB VRAM at fp16. Fits Win-5090 (32 GB) comfortably, **borderline on Linux-3090 (24 GB)** — KV cache for our 100+ token prompts may push it over. Fails Mac M4 (16 GB). For now we ship VibeVoice-1.5B (3B total, ~7 GB VRAM) as the cross-rig mid-tier addition; Large 9B is the quality-ceiling future enhancement, gated on either (a) confirming the 3090 fits the model with our prompts or (b) shipping a 4-bit quantized GGUF variant (community has `FluffyBunnies/vibevoice-onnx-int4` and `callgg/vibevoice-bf16` as candidates).

- **[Fish Audio S2 / S2-Pro](https://github.com/fishaudio/fish-speech)** (current `main` branch, 4B params). **Reclassified 2026-05-24 from skipped to queued for the Linux-3090 pass.** Prior block was "Linux/WSL only per official docs; Windows native unsupported" — that's resolved on a native Linux rig. 24 GB VRAM floor barely fits the 3090. Still needs a Python wrapper to orchestrate the 3-stage CLI pipeline (DAC encode → text2semantic → DAC decode) so the runner sees `text in → wav out`. Wrapper is ~1-2 days of work. Will run on Linux-3090 only; Win + Mac rows show fail (documented). Research-license non-commercial (benchmark redistribution OK).

- **[MOSS-TTS-Realtime](https://github.com/OpenMOSS/MOSS-TTS)** (OpenMOSS, 1.7B, `MossTTSRealtime` architecture, Apache 2.0). Streaming/real-time TTS for low-latency voice agents — upstream advertises a TTFB of 180 ms via incremental synthesis with multi-turn context. Needs a streaming-bench lens (sub-sentence latency, time-to-first-chunk, mean inter-chunk gap) which our current harness doesn't measure. A non-streaming wrapper would understate the model's design point — the 180 ms claim only shows up when the harness watches the first audio frame rather than the full tensor. Queued behind the "streaming bench lens" future enhancement (see [tasks.md](tasks.md)). The non-realtime siblings MOSS-TTS-Nano (100M) + MOSS-TTS flagship (8B) are in the harness as of 2026-05-25.

- **[MOSS-TTSD / MOSS-VoiceGenerator / MOSS-SoundEffect](https://github.com/OpenMOSS/MOSS-TTS)** (OpenMOSS, 1.7B-8B, Apache 2.0). Three sibling models in the MOSS-TTS family with different task shapes:
  - **MOSS-TTSD** is a *dialogue* model (multi-speaker, ultra-long dialogues). The harness's `(text in → wav out)` contract assumes single-speaker monologue prompts; benching TTSD against single-speaker models would compare apples to oranges. Closer fit: a Dia-style "dialogue bench" track once one exists.
  - **MOSS-VoiceGenerator** is a *voice design* model — generates voices from text prompts without any reference speech. Task shape is "describe a voice → produce a speaker embedding/wav", orthogonal to either predefined or cloning TTS.
  - **MOSS-SoundEffect** generates sound effects, not speech. Out of scope for a TTS bench.
  All three deferred until a separate task-shape track exists for each.

## Skipped (still excluded)
- **[Fish Audio S1-mini](https://huggingface.co/fishaudio/s1-mini)** (0.5B distilled S1). Small enough to fit in principle, but the S1 inference code lives at a specific mid-2025 *commit* (no tag) — pre-S2 branch's `v1.5.1` doesn't pair (different `firefly-gan-vq-*` filenames), and `v2.0.0-beta` is S2-only. Pinning to commit `781bf1cd` works, but pulls a heavy dep tree (lightning, wandb, gradio, faster_whisper, modelscope, funasr, `pyaudio`) and still needs the same 3-stage CLI subprocess wrapper. CC-BY-NC-SA license. Worth revisiting if the user wants the model specifically — the install path just doesn't pay for itself in a "fast comprehensive bench" context.
- **[Orpheus TTS](https://github.com/canopyai/Orpheus-TTS)** (Canopy Labs, 3B Llama-based). High-quality emotional/empathetic speech with ~200ms streaming latency, but the inference path is `pip install orpheus-speech` which depends on **vllm**. vllm has no first-party Windows wheel; community Windows wheels exist (SystemPanic, devnen) but the Blackwell-compatible one (5090 / sm_120) is pinned to vllm 0.20.0 + cu132 while Orpheus pins vllm 0.7.3 — version mismatch is unrecoverable without rebuilding from source. Linux/WSL or Mac (via LM Studio + GGUF) work, but this is a Windows-primary bench so Orpheus is out of scope here. Revisit on the Mac M4 Pro pass.
- **[CosyVoice 3](https://github.com/FunAudioLLM/CosyVoice)** (FunAudioLLM, Fun-CosyVoice3-0.5B-2512). High-quality multilingual cloning + 18 Chinese dialects, but installs as a source clone (no pip wheel) with a researcher-pinned `requirements.txt`: `torch==2.3.1` from a cu121 extra-index (incompatible with Blackwell sm_120 / 5090 — would need a post-install swap), `onnxruntime==1.18.0` pinned in a way uv resolves against a Linux-only CUDA wheel feed first, plus 40+ other exact-version pins (`gradio==5.4.0`, `lightning==2.2.4`, `openai-whisper==20231117`, `transformers==4.51.3`, `tensorrt-cu12` Linux-only). Same family of issue as fish-speech and Orpheus: not a clean cross-platform install. Doable with significant install-archaeology effort — revisit if/when the upstream ships a Windows-friendly requirements set.
- **[Higgs-Audio v3 TTS](https://huggingface.co/bosonai/higgs-audio-v3-tts-4b)** (Boson AI, ~4B). **Reclassified 2026-06-04 from skipped to INCLUDED as the harness's first server-backed model (Linux-3090).** Background: v2 (`bosonai/higgs-audio-v2-generation-3B-base`) stayed blocked because the installable `boson_multimodal` package only registers the v1 `HiggsAudioModel` arch, and the v2 DualFFN weights won't load into it. v3 doesn't fix that in-process either — the v3 checkpoint ships only `config.json` + `tokenizer.*` + `model.safetensors` (no `modeling_*.py`, no `auto_map`), the `higgs_multimodal_qwen3` class isn't in stock transformers, and raw `.generate()` would emit audio *tokens* that still need the Higgs tokenizer decoder. So there is no single-process Python path. The one supported path is a Docker container running `sgl-omni serve` exposing OpenAI-style HTTP `/v1/audio/speech`; the runner (`runners/higgs_v3_runner.py`) is a thin HTTP client. Tym OK'd relaxing the old "no Docker / no server" rule for this on Linux (2026-06-04). 24 kHz, 100 languages, zero-shot in-context cloning. License = "Boson Higgs Audio v3 Research and Non-Commercial License" — explicitly permits non-production benchmarking/evaluation by commercial entities and publication of benchmark outputs / sample clips in non-commercial repos (passes the redistribution bar). HF-gated download (accept terms + HF_TOKEN). Linux-only / CUDA-only; see [known-issues.md](known-issues.md) for the server-standup gotcha. The v2 entry stays unregistered (no v2 class in the package); v3 supersedes it.
- **[Voxtral 4B TTS](https://huggingface.co/mistralai/Voxtral-4B-TTS-2603)** (Mistral AI, 4B, 9 langs, 20 preset voices). Same vLLM-on-Windows-Blackwell wall as Orpheus: the HF model card declares `library_name: "vllm"` and the file layout is Mistral's custom format (`consolidated.safetensors`, `params.json`, `tekken.json`) — no `config.json` / `tokenizer.json`, so `transformers.AutoModel` can't load it at all. Inference is only documented via vLLM-Omni (vllm >= 0.18.0). Linux/WSL would work, and there's a community MLX port for Mac (`jason1966/Voxtral-TTS-MLX`), but the Windows-primary bench is out of scope. Revisit on the Mac M4 Pro pass via the MLX port, or when a Linux/WSL pass is added.
- **[Step-Audio-TTS-3B](https://huggingface.co/stepfun-ai/Step-Audio-TTS-3B)** (StepFun AI, 3.5B). Two blockers stacked: (1) bundles Linux-only compiled C++ extensions (`lib/liboptimus_ths-torch2.X-cu12X.cpython-310-x86_64-linux-gnu.so`) that `modeling_step1.py` imports at runtime via `trust_remote_code` — no Windows or Mac equivalents shipped. (2) The upstream github repo linked from the model card (`stepfun-ai/Step-Audio`) returns 404 — appears abandoned/renamed. The successor (`stepfun-ai/Step-Audio2`) is a different project (audio understanding LLM, not TTS). With no current install docs and Linux-only native deps, this is effectively a Linux-only model. Revisit on a Linux/WSL pass, or if StepFun publishes a cross-platform inference path.
