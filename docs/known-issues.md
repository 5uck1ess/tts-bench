# Known issues

Frictions surfaced while building the harness. None are blockers on Mac/Linux — most apply to Windows or to specific models.

## Cross-cutting

- **`uv venv` doesn't seed pip.** `python -m pip install ...` fails inside a fresh uv venv. Installs use `uv pip install --python <venv-python>` throughout. The install scripts codify this.
- **PowerShell 5.1 + native exes + `$ErrorActionPreference=Stop` is a trap.** `uv` prints to stderr on success (`"Using CPython 3.11.15"`); PowerShell wraps each stderr line as a `NativeCommandError` and trips Stop even on exit 0. `install.ps1` uses `$LASTEXITCODE` checks instead.

## Per-model

- **Miso TTS 8B — cloning is unstable shot-to-shot (benched at cooler sampling), and output texture is artifact-prone in BOTH modes.** At upstream sampling defaults (temperature 0.9 / topk 50) the clone usually loses the reference voice entirely — the by-ear A/B (2026-06-10, chris reference) failed at 0.9/50 and held the voice at 0.7/30 and 0.5/20, with one 0.7/30 generation still drifting. The runner benches cloning at **temp 0.7 / topk 30** (default-voice mode keeps upstream sampling), but voice retention remains stochastic per generation — flagged `⟳ clone unstable` (warn) on the Listen gallery; per-clip SIM against the reference ranges **0.03–0.74**. Separately, by-ear QA hears **audible artifacts/glitches in both modes** — the default voice *character* is good but the texture is rough — flagged `⚠ artifacts` (warn) on the default lens. The objective scores agree: UTMOS ~3.9 (bottom half) and mean WER ~0.42 (3rd worst; long prompts sometimes run on to the 30 s generation cap). Caveat: both rigs run our torch 2.7.1 + torchtune 0.6.1 stack in place of upstream's pinned torch 2.4.0 + torchtune 0.4.0 (pre-Blackwell), so a stack contribution to the artifacts isn't ruled out — the hosted demo at misolabs.ai is the stack-independent reference.
- **OmniVoice MPS — long-prompt "OOM" was a spurious MPS-watermark trip (FIXED for default voice).** The 30-word Parakeet paragraph (prompt 3) failed with "MPS backend out of memory (MPS allocated: 3.38 GiB, other allocations: 16.73 GiB)" on a 16 GB M4 — but the allocator was over-counting unified-memory "other allocations": real usage is only ~1.1 GB. **Fix:** `omnivoice_runner.py` sets `PYTORCH_MPS_HIGH_WATERMARK_RATIO=0.0` (disabling the early-OOM ceiling) before importing torch on the mps device; the long-form default-voice cell now completes at ~0.4–0.7× RTF. **Cloning** (with the canonical `chris_hemsworth_15s.wav` reference) also benefits from the same watermark fix: mps now passes **4/5 prompts** — only the 30-word long-form (prompt 3) genuinely pushes past 16 GB and is OS-killed (SIGKILL); cpu does the full **5/5**. Both are sub-realtime (~0.1–0.3× RTF), so OmniVoice cloning on a 16 GB Mac works but is slow — prefer a CUDA rig for cloning throughput.
- **OmniVoice — tops the blind cloning vote on voice match, but intelligibility is its weak spot (don't read #1 as "best overall clone").** The pairwise study (`docs/cloning.md`) ranks OmniVoice #1 because listeners pick the better **voice/accent match** — and its timbre fidelity genuinely is top-tier. But that 2AFC doesn't measure intelligibility, and OmniVoice can **drop or garble words** (high WER), so a clip can win the A/B while being less usable than a lower-ranked, fully-intelligible clone. The README/`cloning.md` claims are qualified accordingly, and the objective **WER** column on the [Scores lens](https://5uck1ess.github.io/tts-bench/scores.html) exists to quantify exactly this — audition (or check WER) before relying on OmniVoice for anything where the words have to land.
- **VibeVoice 1.5B — not viable on a 16 GB Mac (GPU-class here).** With the reference wav present it *does* run on **cpu**, but at ~0.03–0.07× RTF (67–87 s to generate 2–5 s of audio) and the long-form prompt times out at the 600 s cell limit; on **mps** it OOMs at load (`RuntimeError: Invalid buffer size: 10.94 GiB` — the 1.5B in float32 exceeds 16 GB). It is therefore tagged GPU-class (`harness.py`) and skipped on non-CUDA rigs by default (force with `bench.py --all`). It runs normally on the CUDA rigs. The 0.5B streaming variant (`vibevoice`) runs fine on both cpu and mps. For the 1.5B use CUDA or a 32 GB+ machine.
- **VibeVoice 1.5B — needs a local 15s reference wav for its default voice.** Unlike the 0.5B variant (which auto-downloads `.pt` voice presets to `~/.cache/vibevoice-voices/`), the 1.5B path ships no bundled voice and falls back to `reference/chris_hemsworth_15s.wav`. That file is gitignored and not shipped, so on a fresh machine the predefined-voice run fails with `FileNotFoundError: Voice reference wav not found`. Drop any ~15s wav at that path, or pass `--reference <wav>`.
- **MARS5 — sub-realtime on Apple Silicon CPU; times out in the bench.** mars5 (torch.hub) on M4 CPU exceeds the 600s per-cell bench timeout and records as `timeout 600s`. It has no MPS path (cpu/cuda only) and is slow even on CUDA (see Pending work in the README). Treat as unusable on Mac.
- **LuxTTS — blocked on Windows AND Apple Silicon.** Depends on `piper-phonemize` 1.1.0 which only ships wheels for `manylinux_2_28_{x86_64,aarch64}` and `macosx_10_14_x86_64` (Intel Mac only). On arm64 macOS (M-series) the install fails with "no wheels with a matching platform tag (e.g., `macosx_26_0_arm64`)". Workaround: build piper-phonemize from source (untested) or use Linux. The README previously claimed "should install cleanly on macOS — piper-phonemize macOS wheels exist" — that's only true for Intel Macs. Also note the cloned repo's pyproject lists `name = "Zipvoice"` even though the GitHub repo is named LuxTTS, so the runner imports `from zipvoice.luxvoice import LuxTTS`. (Piper proper is unaffected — `piper-tts` 1.4+ bundles espeak-ng and dropped the piper-phonemize dependency.)
- **NeuTTS — `pip install neutts` gives the torch backbone, not the production fast path.** Torch backbone on x86 CPU runs at ~0.2× RTF (unusable). Production path is `llama-cpp-python` + `neuphonic/neutts-*-q4-gguf` models. Post-switch RTF is what's in the table. Install scripts install the GGUF path by default.
- **NeuTTS — reference voices need both `.wav` AND `.txt` (transcript).** Wav-only fails inside `encode_reference()`. Pocket-TTS in contrast takes a voice name string or a single wav (gated).
- **NeuTTS Nano — multilingual = separate model file per language.** `neuphonic/neutts-nano-q4-gguf` (EN), `neuphonic/neutts-nano-french-q4-gguf` (FR), etc. Runner switches based on `--language`.
- **Pocket-TTS — voice cloning is HF accept-terms gated.** Predefined voices work without auth. Reference-wav cloning triggers a fetch of the gated `kyutai/pocket-tts` repo and 401s. Either accept terms or use NeuTTS Air/Nano for BYO-voice without auth.
- **Kokoro — misaki tokenizer calls `spacy.cli.download()` at init.** Tries to install `en_core_web_sm` via pip; fails in uv venvs. Install scripts pre-install the model wheel directly to bypass this.
- **KittenTTS — needs `espeakng-loader` (bundles espeak-ng DLL).** System espeak install also works on Mac/Linux but is more friction. Runner sets `ESPEAK_DATA_PATH` env var because the bundled DLL has a hardcoded CI build path.
- **ChatterBox — needs `setuptools<80`.** The `perth` audio watermarker imports `pkg_resources` (removed in setuptools 80+). Install scripts pin the version.
- **F5-TTS — `torchaudio.load()` routes through `torchcodec` in torch 2.12+**, which needs FFmpeg shared DLLs (libtorchcodec_core4.dll etc., NOT just ffmpeg.exe). On Windows with the typical static FFmpeg build this fails. Runner monkey-patches `torchaudio.load` to use `soundfile` directly. Install scripts also pin `datasets<3.0` to avoid pulling torchcodec into the import chain.
- **Fish Speech 1.5 — two Apple-Silicon install gotchas (both fixed in `install.sh`).** (1) **portaudio:** the editable `fish-speech` install pulls `pyaudio`, which compiles against `portaudio.h`. macOS ships none, so the build dies with `'portaudio.h' file not found`. Fix: `brew install portaudio`; `install.sh` now puts its `--prefix` `include`/`lib` on `CPATH`/`LIBRARY_PATH` (and warns if portaudio is missing). (2) **torch cu128 reinstall:** `install.sh` reinstalled torch from `--index-url .../whl/cu128` as its last step, but that index ships **no macOS-arm64 wheel** — uv resolves "unsatisfiable" (`no wheels with a matching platform tag, e.g. macosx_*_arm64`) and the install aborts. Fix: the cu128 reinstall is now Darwin-guarded (Mac keeps the default torch the editable install already pulled), matching styletts2/zonos/openvoice. Post-fix, fish_15 benches on Mac but is sub-realtime (cpu ~0.34×, mps ~0.31× RTF — it's a GPU-class model here).
- **VibeVoice — install from `vibevoice-community/VibeVoice`, NOT pypi or `microsoft/VibeVoice`.** pypi `vibevoice==0.0.1` only ships the base architecture (no streaming class). The official Microsoft repo was taken down in September 2025 then partially restored without code. The community fork at github.com/vibevoice-community/VibeVoice keeps the original code and added a working streaming variant on 2025-12-04. Voice presets (`.pt` files, 2-4MB each) are not bundled in the package — runner auto-downloads them from the fork to `~/.cache/vibevoice-voices/` on first use.
- **VibeVoice — the "you should probably TRAIN this model" warning is benign.** The HF checkpoint deliberately omits the `acoustic_tokenizer.encoder` weights (~400 keys load as random). That subnet is unused at inference because the `.pt` voice presets are pre-encoded representations. Audio output is unaffected.
- **ZipVoice — needs `torchcodec` + FFmpeg shared libs to load the reference wav.** The venv's torch 2.11+ routes `torchaudio` audio loading through `torchcodec`; without it, reference loading fails with `ImportError: TorchCodec is required for load_with_torchcodec`. Install scripts add `torchcodec`, and `zipvoice_runner.py` discovers the FFmpeg lib dir and re-execs once with it on `LD_LIBRARY_PATH`, so a non-standard FFmpeg (e.g. linuxbrew, whose libs aren't on the default loader path) is found. torchcodec supports FFmpeg 4–8. Windows needs the "full-shared" FFmpeg build (ships DLLs); alternatively pin torch<2.9 to skip torchcodec. (F5-TTS dodges the same issue differently — it monkey-patches `torchaudio.load` to use `soundfile`.)
- **ZipVoice — CUDA OOMs on ≤24 GB GPUs (k2 not installed).** k2 is intentionally skipped (no wheels for the venv's torch/CUDA combo), so the Swoosh ops fall back to a memory-hungry PyTorch path that allocates >24 GB of VRAM and OOMs even on short prompts (e.g. an RTX 3090 fails on a 20 MiB request because the process already holds ~23.5 GB). CPU works fine — the same path uses ~30 GB RAM. Use CPU on ≤24 GB rigs; a 32 GB+ card (e.g. RTX 5090) fits it.
- **MOSS-TTS-Nano (100M) — works everywhere; MOSS-TTS flagship (8B, v1.0 + v1.5) runs on the Linux-3090.** The Nano variant installs and runs fine (CPU + CUDA). **Install gotcha:** the 8B's `[torch-runtime]` extra is installed with `--extra-index-url cu128`, and uv's default `first-index` strategy then finds `tqdm` on the cu128 torch index but not the pinned `tqdm==4.67.1` (which is on PyPI), and won't fall back, so the solve fails. Adding `--index-strategy unsafe-best-match` lets it resolve `tqdm` from PyPI while torch comes from cu128. **VRAM — fits after all (note corrected 2026-06-08):** an earlier note claimed the 8B Qwen3-backbone wouldn't fit the 3090's 24 GB, but both `moss_tts` (v1.0) and `moss_tts_v15` (v1.5) bench cleanly on the 3090 (bf16 weights ~16 GB; v1.5 runs at 0.9–2.3× RTF — faster than realtime). Both are benched/published from Linux. v1.5 needs an explicit `language=` per prompt (with it omitted v1.5 can mis-route language); the runner passes it.
- **Zonos — `--device cpu` crashes on a CUDA-equipped host (fixed in the runner).** Zonos computes `DEFAULT_DEVICE = get_device()` at *import* time (`zonos/utils.py`), and `get_device()` returns `cuda` whenever any GPU is visible. Its conditioning helpers (`make_cond_dict`, `prepare_conditioning`) default new tensors to `DEFAULT_DEVICE`, so even when the model is loaded with `device="cpu"` the conditioning tensors land on `cuda:0` and `generate()` dies with `RuntimeError: Expected all tensors to be on the same device ... mat1 is on cuda:0 ... other tensors on cpu` (a `wrapper_CUDA_addmm`). It only bites CUDA rigs (Windows/Linux) — on a CPU-only box like a Mac, `get_device()` already returns cpu, which is why the cpu path works there. Fix: `zonos_runner.py` sets `CUDA_VISIBLE_DEVICES=-1` before importing torch/zonos when `--device cpu`, so `get_device()` (and therefore `DEFAULT_DEVICE`) resolves to cpu. Use `-1`, **not** `""` — an empty string is treated as *unset* and leaves all GPUs visible (the cpu cell then still failed with the same `cuda:0` error before the value was corrected).
- **Qwen3-TTS CUDA-graph (`qwentts_fast`) — cloning runaway (FIXED).** The fast engine's default text-feeding (`non_streaming_mode=False`, step-by-step during the CUDA-graph decode) never emits EOS for some longer prompts, so generation ran away to `max_new_tokens` (~147 s of garbled/looping audio; prompts 2–5 hit it stochastically). Default voice was unaffected. **Fix:** the runner now passes `non_streaming_mode=True` (prefill the full target text before decode) — it terminates correctly *and* is ~5–30× faster (gen_s ~1 s vs up to 38 s). Re-benched; durations are now sane (p2: 147 s → 4.3 s).
- **Fish Speech S2-Pro — cloning run used the wrong reference (a bundled default voice, not Chris).** The `fish_s2` runner's no-`--reference` default is `reference/jo.wav` (`{"en": "jo.wav", "fr": "juliette.wav"}`). On the Linux-3090 cloning pass it didn't receive a usable `--reference` and fell back to that default — so the cloning sample imitates jo/juliette, not the `chris_hemsworth` target every other model used. (`step_editx` shares the same default but cloned Chris correctly on this pass, and `metavoice` defaults to the 67 s `chris_hemsworth.wav`, so both of those are fine — only `fish_s2` needs fixing.) **Fixed:** re-benched on Linux with `--reference reference/chris_hemsworth_15s.wav` and `merge.py --replace`d into `linux-cloning`; it now clones Chris like every other model.
- **Higgs Audio v3 TTS — the bench's first SERVER-BACKED model; Linux + CUDA only, needs a Docker server running before you bench it.** v3 (`bosonai/higgs-audio-v3-tts-4b`, ~4B, `higgs_multimodal_qwen3`) ships only `config.json` + `tokenizer.*` + `model.safetensors` — no `modeling_*.py`, no `auto_map` — and the class isn't in stock transformers, so there's no single-process Python load (raw `.generate()` would emit audio *tokens* that still need the Higgs tokenizer decoder, wired only by SGLang-Omni). The supported path is a Docker container running `sgl-omni serve` exposing OpenAI-style HTTP `/v1/audio/speech`; `runners/higgs_v3_runner.py` is a thin HTTP client (venv = `requests`+`soundfile`+`numpy`, **no torch**). Standup: `docker pull lmsysorg/sglang-omni:dev`, then `docker run … --gpus all -v "$(pwd)":"$(pwd)" -w "$(pwd)" -e HF_TOKEN=hf_… lmsysorg/sglang-omni:dev`, inside it `uv pip install -e .` the cloned `sglang-omni`, accept the HF terms, and `sgl-omni serve --model-path bosonai/higgs-audio-v3-tts-4b --port 8000` (full recipe in `install.sh`'s higgs_v3 stanza). Gotchas: (1) the bench's normal cuda probe imports torch *in the venv*, which this venv lacks — the harness special-cases server-backed models (`SERVER_BACKED` set) to probe the GPU via `nvidia-smi` instead, so the cuda cell builds on a CUDA rig and is skipped (gpu-class) on non-CUDA rigs. (2) Cloning passes the reference by file path (`references[].audio_path`); the `-v "$(pwd)":"$(pwd)"` bind-mount makes the repo's real path valid inside the container so the absolute path resolves (a base64 inline form is preferred if a future cookbook adds it — see `_REF_USE_BASE64`). (3) `peak_vram_mb` is `None` (the model lives in the container, not the runner process); a coarse whole-GPU `gpu_used_mb` is reported via `nvidia-smi` but includes the whole server. (4) First pass is non-streaming, so `ttfa_ms == gen_s` (Higgs advertises sub-second TTFA via the streaming SSE endpoint — a later enhancement). 24 kHz, 100 languages.
- **Higgs Audio v3 — the default voice is sampled fresh each generation, so it's a different speaker per prompt.** With no `--reference`, the server's `voice="default"` draws a speaker from the model's voice prior, and the runner samples at `temperature=0.8`/`top_k=50`, so each of the 5 default-voice clips is a *different* identity (this is genuine synthesis, not a fallback Chris clone — so it stays in the Default-voice gallery, unlike the eleven no-default models below). **Cloning** (with `ref_audio`+`ref_text`) conditions on the reference and stays consistent across prompts. Surfaced as a ⚠ badge on its Default row in the Listen gallery. To pin one consistent default speaker, add a fixed `seed` to the runner's default-voice payload and re-bench the default pass.
- **DramaBox — CUDA-only (~18 GB VRAM), expressive *dialogue* TTS on an LTX-2.3 video-model backbone; short prompts get heavy silence padding (trimmed in the runner).** DramaBox (`ResembleAI/Dramabox`, 3.3B audio-only DiT, IC-LoRA fine-tune of LTX-2.3) is a source-clone install (no PyPI): the runner imports `src.inference_server.TTSServer` from `venvs/dramabox/src`, fetches the DiT + audio-components + a 4-bit Gemma-3-12B text encoder via the repo's `model_downloader` (~16 GB, HF-cached), and builds the server with explicit `checkpoint`/`full_checkpoint` paths + `compile_model=False` (the bare `TTSServer()` defaults to non-existent local `models/` files, and torch.compile/Triton is brittle on Windows). bitsandbytes (the 4-bit Gemma) *does* ship a working Windows wheel (0.49.2 on the 5090). Peak VRAM ~18 GB (audio-only mode frees LTX's video stack) → fits both the 5090 and the 3090. 48 kHz out (LTX "BWE" bandwidth-extension decode). Prompt grammar is `<speaker description>, "<dialogue>" <action>`; the runner wraps plain bench prompts in a neutral `A person speaks clearly, "…"`. Watermarked (Resemble Perth). **Gotcha:** DramaBox's duration estimator over-allocates for short lines, so a clip can open with several seconds of dead air — the runner trims leading/trailing silence (energy < −40 dB rel. peak, 0.1 s pad) so `audio_s`/RTF aren't inflated. The optional RE-USE voice-ref denoiser (mamba-ssm) is skipped (no Windows wheel; off by default). Both lenses: no-reference = prompt-described voice (Default), 10 s+ wav = cloning.
- **dots.tts — CUDA-only 2B continuous AR TTS; the sampler MUST be seeded or it early-stops to a ~0.3s fragment.** dots.tts (`rednote-hilab/dots.tts`, Apache-2.0) is a pip package (`dots_tts`) source-clone editable install; the runner does `from dots_tts.runtime import DotsTtsRuntime` and `from_pretrained("rednote-hilab/dots.tts-soar", precision="bfloat16")` (the SCA flagship — best Seed-TTS-Eval quality; weights snapshot-download from HF on first run, ~5.3 GB VRAM at bf16, fits the 3090). **Gotcha:** the AR + flow-matching head hits an early end-of-audio if the RNG isn't seeded — an unseeded run emitted only 2 audio patches (15360 samples = 0.32 s) for a full sentence. The CLI/gradio both call `seed_everything()` before generating; the runner mirrors this (`seed_everything(42)` before each `generate`), after which "Open the browser and read my email." renders at the expected ~1.9 s. Sampler params follow the upstream **CLI** defaults (`guidance_scale=1.2`, `num_steps=10`, `speaker_scale=1.5` default) — NOT the README quick-start's `guidance_scale=1.0`, which is below the runtime default. Multilingual (24 langs incl. FR), 48 kHz. **Rig note — LINUX-ONLY.** The Windows install never worked — its `WeTextProcessing`→`pynini` dependency won't build under Windows MSVC (`cl` rejects pynini's GCC/Clang `-Wno-register` flag → `D8021`), and that import is hit at module load (`runtime.py` → `utils.text` → `tn`) so it can't be skipped — so dots.tts has **no `install.ps1` stanza** and is benched/published from the **Linux-3090** only. **Default voice — it has one, but it varies.** dots.tts *does* synthesize with no reference (it's not reference-only), but that own-voice is sampled fresh per prompt — a different speaker each generation, exactly like Higgs v3 — so it's NOT in `NO_PRESET_VOICE`; it shows in the Default-voice gallery with a "⟳ voice varies" badge, and under Cloning (consistent) when given a reference. Revisit `install.ps1` only if a working Windows path is found.
- **Twelve zero-shot models have no "default voice" — their no-`--reference` run clones the bundled reference.** `moss_tts`, `moss_tts_v15`, `moss_tts_nano`, `fish_15`, `fish_s2`, `metavoice`, `openvoice`, `zipvoice`, `zonos`, `vibevoice_15b`, `vibevoice_7b`, `echo` ship no preset/built-in speaker. With no `--reference` they fall back to `reference/chris_hemsworth_15s.wav`, so their "default voice" bench sample is just a (different, non-deterministic take of a) Chris clone — not a real default. The published Listen gallery therefore shows these models **only under Cloning**, never in the Default-voice section. Models that *do* ship a default — `styletts2` (bundled LibriVox speaker), `vibevoice` 0.5B (`en-Emma_woman` preset), ChatterBox, etc. — stay in Default normally.

## Skipped on Apple Silicon (Apple M4, 16 GB)

These models produced no usable output on the Mac M4 bench pass, so they are marked **skipped** on this rig. They stay in the registry — most run on the Windows/Linux CUDA rigs; the gap is Apple-Silicon-specific (no arm64 wheel, MPS memory limits, or no MPS/CPU path at all).

- **luxtts** — **not possible on Apple Silicon.** `ModuleNotFoundError` at load on cpu and mps because `piper-phonemize` never installs: confirmed via `uv` that *no* version of it ships an arm64-macOS wheel (`all versions … no wheels with a matching platform tag, e.g. macosx_*_arm64`) — wheels exist only for Linux (`manylinux_2_28_*`) and Intel Mac (`macosx_*_x86_64`). This is the **same gap as Windows** (no Windows wheel either). Would require building piper-phonemize from source (untested); use Linux instead.
- **mars5** — **not possible (too slow).** Sub-realtime on M4 CPU, exceeds the 600 s per-cell timeout, and has no MPS path (cpu/cuda only). Even with a longer timeout the result would be unusable.
- **vibevoice_15b** — **not viable (GPU-class here).** Runs on cpu with the reference present but at ~0.03–0.07× RTF and long-form times out; mps OOMs at load (~10.9 GiB). Now tagged GPU-class and skipped on non-CUDA by default — see the per-model entry above. The 0.5B variant (`vibevoice`) runs fine on both.
- **dia**, **qwentts_fast** — CUDA-only; torch reports no available device on Apple Silicon, so the harness builds no cells for them here.

Partial — CPU usable, mps skipped: **zipvoice** cloning (cpu works 10/11 once FFmpeg is present — `brew install ffmpeg` so torchcodec can load the reference; mps OOMs ~20 GiB. It has no default voice, so it only appears under Cloning), **omnivoice** cloning on mps (4/5 with the watermark fix — only the long-form prompt 3 is OS-killed >16 GB; cpu does 5/5; both sub-realtime — see above), and **voxcpm** cloning (600 s timeout; default-voice cpu is fine).
